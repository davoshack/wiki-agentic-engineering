# Agentic testing

Testing — automated *and* manual — as the operational layer that turns coding-agent
output from "plausible" into "demonstrably works." The practice combines three things:
seeding sessions with the existing test suite, building new functionality with
test-first (red/green) TDD, and supplementing the suite with manual testing
(direct invocation, `curl`, browser automation, "show your work" artifacts). Each
manual finding feeds back into the automated suite, so the two modes compound.

## Synthesis

### Why testing matters more in the agent era, not less

[[simon-willison]] opens the manual-testing chapter with the strongest restatement yet
of the foundational claim: *"The defining characteristic of a coding agent is that it
can execute the code that it writes ... Never assume that code generated by an LLM
works until that code has been executed."* Code execution is the defining tool of a
[[coding-agents|coding agent]]; tests are the discipline that turns execution into
*evidence*.

The historical excuse for skipping tests — they're expensive to maintain — collapses
when an agent can write or update them in minutes. The historical case *for* tests
strengthens at the same time: AI-generated code that has never been executed is "pure
luck" in production. Tests are also onboarding material for the agent itself; ask
Claude Code about an existing feature and it will almost certainly find and read the
relevant tests first.

In Karpathy's frame ([[verifiability]]), tests are the *canonical reward signal* — the
thing that makes coding such a productive RL substrate. Agentic testing is what wires
that signal into a project so the agent (and the human reviewing its work) can use it
every turn.

### Three modes, one loop

Across [[2026-05-26-first-run-the-tests]], [[2026-05-26-red-green-tdd]], and
[[2026-05-26-agentic-manual-testing]], Willison describes a three-mode workflow:

**1. Seed the session with the existing suite.**

> First run the tests

The four-word prompt that opens a new session against an existing project. It does three
things at once: forces the agent to discover the test harness (so it will re-run tests
later by reflex), shows it the rough test count (a proxy for project size and a pointer
into the suite as documentation), and puts it in a testing mindset for the rest of the
work. For projects with a known harness, a one-step variant: `Run "uv run pytest"`.

**2. Build new functionality test-first with red/green TDD.**

> Use red/green TDD

Test-first development is a *fantastic* fit for coding agents because it constrains the
two characteristic failure modes simultaneously:

- Code that **doesn't work** — failing tests catch it before they pass.
- Code that **nobody needs** — the test demands an implementation; you don't write
  speculative code first.

The "red" step is load-bearing: confirm the tests fail before implementing, or you risk
a test that already passes and never actually exercises the new code. Every good model
reads `red/green TDD` as shorthand for the full discipline.

**3. Manual-test what tests don't catch.**

Even with a comprehensive suite, automated tests miss things — server fails to start,
a UI element doesn't render, an edge case the suite never thought of. Mode-by-mode:

- **Python libraries:** `python -c "...code..."` for direct invocation, often
  multi-line.
- **Compiled or other languages:** write a scratch program in `/tmp` (kept out of the
  repo) and compile + run it.
- **JSON APIs:** run the dev server and let the agent *explore* the API with `curl` —
  "explore" is the operative word; it produces broad coverage.
- **Web UIs:** use Playwright or a dedicated wrapper (agent-browser, Rodney). With
  Rodney, "look at screenshots" reminds the agent to use its own vision capabilities to
  evaluate visual output.

**The loop closes when manual findings flow back into the suite.** Anything caught by
manual testing should be fixed with red/green TDD so the case becomes permanent
regression protection. Manual testing and the automated suite compound rather than
substitute.

### Short prompts as compressed discipline

A pattern visible across this chapter cluster: a *four-word prompt activates a
substantial chunk of pre-existing software-engineering practice that is already baked
into the models.* `First run the tests` and `Use red/green TDD` are the named examples;
the trick generalises. Models internalised TDD, suite seeding, edge-case probing, and
exploratory API testing from years of human text; the prompt is just the key that opens
the right drawer.

This rhymes with [[hoarding-working-examples|"hoard working examples"]] and the *"sort
out this git mess for me"* prompt in [[git-with-agents]]: compact phrases that load a
disproportionate amount of latent capability when used with confidence.

### Tool surface for manual testing

[[2026-05-26-agentic-manual-testing]] surveys the current browser-automation landscape
through an agent-as-user lens:

- **[Playwright](https://playwright.dev/)** — Microsoft's heavyweight, multi-binding,
  multi-engine. "Test that with Playwright" is often enough.
- **[agent-browser](https://github.com/vercel-labs/agent-browser)** (Vercel) — a CLI
  wrapper around Playwright designed specifically for coding agents.
- **[Rodney](https://github.com/simonw/rodney)** (Willison) — uses Chrome DevTools
  Protocol directly. Designed with three deliberate affordances:
  1. `uvx rodney --help` so the agent self-installs on first use.
  2. The `--help` text is written *as agent-onboarding documentation* — what Rodney is
     and how to use it, in one place.
  3. Prompts like "look at screenshots" steer the agent into using vision against the
     captured images.

These are concrete instances of [[agent-native-infrastructure]] — tools whose interface
is designed for an agent reading `--help` rather than a human reading a website.

### Show your work — Showboat and the anti-cheating artifact

[Showboat](https://github.com/simonw/showboat) is Willison's tool for capturing the
*record* of an agentic manual-testing run as a Markdown document. Three commands:

- `note` — append a Markdown note.
- `exec` — record a command, run it, and record its real output. This is the load-bearing
  one: capturing both the command *and* the actual output is designed to discourage the
  agent from writing what it *hoped* had happened.
- `image` — embed an image (a Rodney screenshot, say).

This addresses a real failure mode: agents that report success without having
genuinely verified it. By forcing the output capture to be mechanical rather than
narrative, Showboat removes the seam where an agent can drift from facts. The
resulting document is also the exact form of *evidence-of-work-done* that
[[agent-code-review]] wants in a good agentic PR — notes on manual testing, command
invocations and their outputs, screenshots of the feature working.

The broader idea Willison flags: *agents showing their work.* Demos and documented
experiments are how reviewers confirm the agent comprehensively solved the problem;
the substrate for routinely producing them is now mature.

## Open questions

- How does "First run the tests" interact with very large suites where running the
  whole thing is slow or expensive? When should the agent run a subset instead, and how
  should the human prompt that?
- How well does red/green TDD hold up for non-functional requirements — performance,
  accessibility, security — that are hard to express as a passing test?
- Browser tests have historically been the flakiest part of the suite. Does
  agent-maintained Playwright actually solve the flake problem, or does it just move
  the cost?
- Is "show your work" through tools like Showboat a durable practice that should scale
  to teams, or a Willison-specific habit? What's the team-scale analogue?
- "Vibe coding" definitionally skips review and tests; "agentic engineering"
  definitionally includes them. Is there a productive middle — *test-only-the-load-bearing-bits*
  — for the agent equivalent of throwaway scripts, or is the binary the right framing?
- For founders building on [[verifiability]]'s "valuable, verifiable, undertrained"
  thesis: how do the patterns here generalise from *code* (the canonical verifiable
  domain) to other domains where the agent's actions need to be checked?
